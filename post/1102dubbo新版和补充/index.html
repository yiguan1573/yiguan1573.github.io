踩坑

dubbo连接nginx集群报错

  • 因为我的nacos集群是通过nginx连接的，dubbo配置nacos集群时报错Dubbo client can not supported string message: HTTP/1.1 400 Bad Request，错误配置如下
dubbo:
  registry:
    address: 192.168.10.101:3737?namespace=f4d705e4-4a68-48ff-b073-0974b0c9db36
    group: DEFAULT_GROUP
    parameters.register-consumer-url: true
  application:
    name: dubbo-consumer
  protocol:
    name: dubbo
    port: 20880

正确的配置如下

dubbo:
  registry:
    address: nacos://192.168.10.101:3737?namespace=f4d705e4-4a68-48ff-b073-0974b0c9db36
    group: DEFAULT_GROUP
    parameters.register-consumer-url: true
  application:
    name: dubbo-consumer
  protocol:
    name: dubbo
    port: 20880

Current application will subscribe all services(size:2) in registry, a lot of memory and CPU cycles may be used, thus it’s strongly recommend you using the externalized property ‘dubbo.cloud.subscribed-services’ to specify the services

  • 报错原因：dubbo.cloud.subscribed-services : provider 用于服务消费方订阅服务提供方的应用名称的列表，若需订阅多应用，使用 “,” 分割。不推荐使用默认值为 “*"，它将订阅所有应用，默认订阅所有应用
  • 解决:yml文件配置一下需要订阅服务
dubbo:
  cloud:
    subscribed-services: 'service1,service2'#配置订阅服务多个服务用,隔开,不配置默认订阅注册中心所有服务

dubbo和feign那个快

  • 经过简单的测试两者速度差不多
    • dubbo
    • feign
  • 分析
    • feign基于http,dubbo基于tcp
    • 在一般简单场景下两者差距不大，高并发场景下使用dubbo更快

为什么有了http还要rpc

  • 参考

    • HTTP和RPC区别（通俗易懂超级好）
    • HTTP与RPC的区别

  • 首先两者都是基于socket，rpc基于TCP，http也基于TCP；TCP是70年代出来的，rpc的概念是80年代提出来的，http是90年代出来的

  • 所以应该是为什么有了rpc还要有http，这是为了统一规范，浏览器是B/S架构，必须要统一的标准才能让所有人访问到

  • rpc一般使用长连接池，效率高

  • 在传输的内容方面，http使用的是json，rpc使用的是protobuf或其他体积更小的方式保存结构体数据

  • 一般来说http基于tcp，多一层封包和几次握手校验，性能自然比直接用tcp实现网络传输要差一些，但是http3.0是基于UDP的，性能与RPC的差距不大
