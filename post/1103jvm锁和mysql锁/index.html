  • 使用ReentrantLock

@Service
public class StockService {
    private Stock stock =  new Stock();
    private ReentrantLock lock = new ReentrantLock();
    public void reduce(){
        lock.lock();
        try {
            stock.setStock(stock.getStock()-1);
            System.out.println("库存余量"+stock.getStock());
        }finally {
            lock.unlock();
        }

    }
}

JVM本地锁：三种情况导致锁时效

多例模式

  • 多例模式上锁的对象不相同，导致失效
@Service
@Scope(value = "prototype",proxyMode = ScopedProxyMode.TARGET_CLASS)//多例模式，CGLIB代理
public class StockService {
    @Autowired
    StockMapper stockMapper;

    private ReentrantLock lock = new ReentrantLock();
    public void reduce(){
        lock.lock();
        try {
            Stock stock = stockMapper.selectOne(new QueryWrapper<Stock>().eq("product_code", 1001));
            if(stock != null&&stock.getCount()>0){
                stock.setCount(stock.getCount()-1);
                stockMapper.updateById(stock);
                System.out.println("库存余量"+stock.getCount());
            }
        }finally {
            lock.unlock();
        }

    }
}

事务

@Service
//@Scope(value = "prototype",proxyMode = ScopedProxyMode.TARGET_CLASS)//多例模式，CGLIB代理
public class StockService {
    @Autowired
    StockMapper stockMapper;
    
    private ReentrantLock lock = new ReentrantLock();

    @Transactional
    public void reduce(){
        lock.lock();
        try {
            Stock stock = stockMapper.selectOne(new QueryWrapper<Stock>().eq("product_code", 1001));
            if(stock != null&&stock.getCount()>0){
                stock.setCount(stock.getCount()-1);
                stockMapper.updateById(stock);
                System.out.println("库存余量"+stock.getCount());
            }
        }finally {
            lock.unlock();
        }

    }

}

  • @Transactional注解利用AOP在进入方法前开启事务，方法结束后提交或回滚事务

  • 在一个锁释放后事务提交前，另一个事务获取到锁查询到未来得及更新的库存

  • 隔离级别使用读未提交,再次压测，发现库存变为0了，但是实际生产中不能使用该事务隔离级别

MySQL锁

一条SQL语句完成上述操作

  • 解决三种JVM锁失效的情况
  • 问题
    • 锁范围问题(innodb是行锁)
    • 同一商品有多条库存记录
    • 无法记录库存变化前后的状态

MySQL悲观锁

  • 满足以下条件才是行锁
    • 锁的查询或者更新条件必须是索引字段
    • 查询或者更新条件必须是具体值
  • 使用悲观锁减库存
  • 问题
    • 性能问题（吞吐量与JVM锁差不多，600/s）
    • 死锁问题：对多条数据加锁时，加锁顺序要一致

MySQL乐观锁

  • 依赖时间戳、版本号、CAS机制
  • 使用jmeter发现吞吐量非常小，程序报错栈溢出和连接超时，因为更新操作默认会加锁，后续代码就会一直重试，事务也一直未提交
  • 修改
  • 问题
    • 性能很低
    • ABA问题
    • 读写分离下导致乐观锁不可靠

总结

  • 性能:一个sql >悲观锁>jvm锁>乐观锁
  • 如果追求极致性能、业务场景简单并单不需要记录数据前后变化的情况下。优先选择:一个sql
  • 如果写并发量较低（多读)，争抢不是很激烈的情况下。优先选择:乐观锁
  • 如果写并发虽较高，一般会经常冲突，此时选择乐观锁的话，会导致业务代码不间断的主试。优先选择: mysql悲观锁
  • 不推荐jvm本地锁。
